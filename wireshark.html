$ wireshark — packet analyzer

Capturing and inspecting the raw data traveling across a network

# What is a Packet? (With an Analogy)

Every piece of information sent over a network is broken into small, manageable chunks called packets. A single webpage might be split across dozens or hundreds of packets.

Here's the analogy that made packets click for me: Imagine you need to mail a 500-page book to a friend, but you can only send it using standard envelopes that each hold about 5 pages. You'd need 100 envelopes. You'd number each envelope (1/100, 2/100, etc.), write the destination address on each one, and also include a return address so your friend knows who sent it.

Each envelope is a packet. The destination address is the destination IP address. The return address is the source IP address. The envelope number is the sequence number (so the receiver can put the pages back in order). And just like your friend could read the outside of the envelope without opening it, anyone on the network path can see where packets are going—and if the contents aren't sealed (unencrypted), they can read those too.

# What Wireshark Lets You See

Wireshark is a packet capture and analysis tool. It puts your network interface into "promiscuous mode," which means it listens to all traffic it can see on the network segment, not just traffic addressed to your computer. Then it presents that data in a human-readable format, breaking down each packet layer by layer.

During a security investigation or penetration test, Wireshark is invaluable for:

  • Seeing what's actually happening: Is that suspicious device really just sending DNS queries, or is it beaconing out to a command-and-control server?
  • Capturing credentials: Unencrypted protocols like HTTP, FTP, and Telnet transmit usernames and passwords in plain text. Wireshark shows exactly what an attacker would see.
  • Troubleshooting network issues: Are packets being dropped? Is there excessive retransmission? Is something flooding the network?
  • Analyzing malware behavior: In a controlled lab environment, security analysts run malware and capture its network traffic to understand what it's trying to do.

# Real-World Example: Capturing an HTTP Login

Scenario: Unencrypted Website Login

A user visits a website that uses HTTP (not HTTPS) and submits a login form. Here's what an attacker on the same network would capture with Wireshark:

No. Time Source Destination Protocol Info
45 12.438921 192.168.1.50 93.184.216.34 HTTP POST /login.php HTTP/1.1

Packet Details (Frame 45)

Frame 45: 542 bytes on wire (4336 bits)
Ethernet II, Src: IntelCor_12:34:56, Dst: Cisco_ab:cd:ef
Internet Protocol Version 4, Src: 192.168.1.50, Dst: 93.184.216.34
Transmission Control Protocol, Src Port: 49152, Dst Port: 80
Hypertext Transfer Protocol
    POST /login.php HTTP/1.1\r\n
    Host: example.com\r\n
    Content-Type: application/x-www-form-urlencoded\r\n
    Content-Length: 38\r\n
    \r\n
    [Full request URI: http://example.com/login.php]
    [HTTP request 1/1]
    [Response in frame: 47]
    File Data: 38 bytes

Line-based text data: application/x-www-form-urlencoded
username=john.doe@email.com&password=Summer2025!

What This Capture Reveals

Every piece of information in the login request is visible in plain text. The attacker now has:

  • A valid email address (john.doe@email.com)
  • A working password (Summer2025!)
  • The exact login endpoint (/login.php)
  • The user's IP address (192.168.1.50)

The critical lesson: This is why HTTPS (HTTP over SSL/TLS) is non-negotiable for any site that handles sensitive information. With HTTPS, the attacker would only see that the user connected to example.com on port 443—the actual request, including the username and password, would be an encrypted blob of meaningless characters.

In our lab, we captured our own traffic on the local network and saw this exact scenario play out with a deliberately insecure test website. Seeing my own password appear in Wireshark's packet details was a visceral lesson in why encryption matters.

> Wireshark Display Filter Cheat Sheet

When analyzing large packet captures, display filters help you find what matters:

http Show only HTTP traffic
tcp.port == 22 Show only SSH traffic (port 22)
ip.src == 192.168.1.100 Show traffic from a specific source
http.request.method == "POST" Show only HTTP POST requests (where data is sent)
dns Show only DNS queries
tcp.flags.syn == 1 && tcp.flags.ack == 0 Show only connection initiation attempts
