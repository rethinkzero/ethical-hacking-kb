$ ssh — secure shell

Encrypted remote access that replaced dangerous clear-text protocols

# What SSH Does and the Problem It Solved

SSH creates an encrypted tunnel between two computers, allowing you to securely log into a remote system and execute commands as if you were sitting at its keyboard. Everything you type and everything the remote system sends back is scrambled in transit—anyone intercepting the traffic sees only meaningless encrypted data.

Before SSH existed, administrators used Telnet and rlogin for remote access. These tools sent everything in plain text, including usernames and passwords. On an unswitched network (common in the 90s), any machine could see all the traffic. A single compromised computer or a packet sniffer on the network meant every login credential was exposed. SSH was created in 1995 specifically to address this devastating security flaw.

Telnet (Legacy, Insecure)

  • No encryption—passwords sent in plain text
  • No verification of server identity
  • Vulnerable to packet sniffing
  • Should never be used on modern networks

SSH (Modern, Secure)

  • End-to-end encryption
  • Server authentication via host keys
  • Optional public key authentication (no passwords)
  • Can tunnel other protocols securely

# What You Can Do Over an SSH Connection

Once I've established an SSH connection to a remote system, I have the same capabilities I'd have sitting at that machine's physical terminal. In the context of ethical hacking and system administration, this includes:

  • Execute commands remotely: Run any command the remote system allows, from checking system status with top to editing configuration files with nano.
  • Transfer files securely: Using SCP (Secure Copy) or SFTP (SSH File Transfer Protocol), which run over the same encrypted connection.
  • Forward ports (tunneling): Redirect traffic from a local port through the SSH connection to a remote network—useful for accessing services behind firewalls.
  • Execute graphical applications: With X11 forwarding, I can run a GUI application on the remote server and have it display on my local machine.
  • Mount remote directories: Using SSHFS, I can treat a remote directory as if it were a local drive.

# The Lab Command: Connecting to the Raspberry Pi

student@kali:~
$ ssh student@192.168.1.100
The authenticity of host '192.168.1.100 (192.168.1.100)' can't be established.
ECDSA key fingerprint is SHA256:7xKp2M3nQ9rL5vW8yZ1aB4cD6eF0gH2iJ4kL6mN8oP.
Are you sure you want to continue connecting (yes/no/[fingerprint])? yes
Warning: Permanently added '192.168.1.100' (ECDSA) to the list of known hosts.
student@192.168.1.100's password: ********

Linux raspberrypi 5.10.103-v7l+ #1529 SMP Tue Mar 8 12:24:00 GMT 2022 armv7l

Last login: Mon Dec 15 09:23:42 2025 from 192.168.1.50
student@raspberrypi:~ $

Command Anatomy: Breaking Down Every Part

ssh The command that invokes the OpenSSH client. This is the program that initiates the connection.
student The username I'm trying to log in as on the remote system. The Raspberry Pi was configured with a "student" account during lab setup.
@ A separator symbol. Everything before it is the username; everything after it is the host address.
192.168.1.100 The IP address of the remote system. I found this earlier in the lab using nmap to scan the local network for the Raspberry Pi.

What Happened After I Pressed Enter

  1. Host key verification: SSH asked if I trusted this server's fingerprint. This prevents "man-in-the-middle" attacks where someone intercepts my connection and pretends to be the real server. Since this was my first time connecting, I said yes, and SSH stored the fingerprint for future verification.
  2. Password prompt: I entered the password for the student account. The characters weren't displayed as I typed—this is a security feature that prevents shoulder-surfing.
  3. Connection established: I saw the Raspberry Pi's welcome banner showing its operating system and kernel version, then I was given a command prompt on the remote system.

What I did next in the lab: Once connected to the Raspberry Pi, I navigated to a web directory, found a file containing grade data, and modified it to demonstrate how an attacker with SSH access could alter sensitive information. This exercise drove home why SSH credentials must be protected and why default passwords should never be left unchanged.
